# Fluxora SE Copilot Demo — Design Spec

Date: 2026-07-16
Status: Approved (brainstorming phase)

## 1. Purpose

Design a demo showing how a ChatGPT Enterprise workspace — with a custom
Agent, Skills, and a CRM Connector — helps a Sales Engineer (SE) at a
digital-native, aggressively-scaling infrastructure company do their job.

Scope for this phase: produce (a) the business narrative, (b) a CRM data
model, (c) a Python script that generates realistic synthetic data for that
model, and (d) a guideline document explaining how to configure the
ChatGPT Enterprise side (Agent, Skills, Connector). **No live system
integration is built** — this is data + documentation only.

## 2. Narrative & Business Context

**Company: Fluxora** (fictional) — a data/AI infrastructure platform
(lakehouse storage, pipeline orchestration, ML model serving), positioned as
a smaller, faster-growing peer to Databricks/Snowflake. ~$400M ARR, ~1,100
employees, late-stage growth stage.

**Target buyer**: engineering and data leadership at mid-market/enterprise
companies migrating off legacy data warehouses or building AI/ML pipelines —
typically a Head of Data Engineering, VP of Engineering, or Chief Data
Officer evaluating a multi-quarter platform commitment.

**Fluxora's Sales Engineering org**: 42 SEs across three pods (Enterprise,
Strategic/named accounts, Mid-Market), reporting through 2 SE managers and a
Director of Sales Engineering. Deal cycles run 90–180 days; the team runs
60–80 active POCs per month.

**Why the problem matters**: The SE team roughly tripled in under two years
to keep pace with company growth, but process didn't scale with headcount.
Specific symptoms:

- **Knowledge silos between pods** — an SE in one pod has no visibility into
  how a peer in another pod handled a similar competitive or technical
  situation.
- **RFP/security-questionnaire volume has outpaced headcount** — SEs spend
  hours rewriting answers that already exist in an old doc or a Slack
  thread.
- **Deal context lives outside the CRM** — why a POC stalled, what a
  customer's real blocker is — in Slack DMs and individual SEs' heads, not
  in structured records. Context is lost when a deal changes hands.
- **SE bandwidth is leadership's #1 cited bottleneck on sales velocity** —
  POCs queue up waiting for SE time that should go to judgment calls, not
  repetitive drafting and summarizing.

**Demo thesis**: A ChatGPT Enterprise "SE Copilot" agent — with skills for
Discovery Prep, POC Tracking, RFP Drafting, and Deal Briefs, backed by a CRM
connector — gives every SE, regardless of pod or tenure, fast access to
institutional knowledge and drafting help, reducing the administrative tax
on their day.

## 3. ChatGPT Enterprise Architecture

**Platform choice**: build the SE Copilot as a **ChatGPT Enterprise
Workspace Agent** (not a classic Custom GPT). Rationale: Agents support
first-class **Skills** as distinct sub-capabilities and can pair with a
**Connector** for live system access, which is a closer match to "agents and
skills/plugins" than a Custom GPT — a GPT can use either an Action or an
App/Connector, but not both, and has no first-class Skill concept. Custom
GPTs + Actions remain documented as a simpler fallback for workspaces where
Agents/Skills aren't yet enabled.

> Note: ChatGPT Enterprise's Agent/Skill/Connector feature set is new and
> actively evolving. Menu names and exact capabilities may shift; the
> guideline doc should be read as directionally accurate, not a pinned
> reference.

**Agent: "Fluxora SE Copilot"**, with four Skills, all reading from one CRM
Connector:

| Skill | Trigger input | Reads from CRM | Produces |
|---|---|---|---|
| Discovery Prep | Account name or Opportunity ID | Account, Contact, past SE_Activity__c | Company background, likely pain points, suggested discovery questions, relevant battlecard pointers |
| POC Tracker | SE name / pod / "my POCs" | POC__c, Opportunity | At-risk POC flags (stalled, missing success criteria), draft status update for AE/manager |
| RFP Drafting | Pasted question or uploaded questionnaire | RFP_Response__c, knowledge files | Drafted answers in Fluxora's voice, gaps flagged for SME input |
| Deal Brief | Opportunity ID | Opportunity, OpportunityTeamMember, SE_Activity__c, Competitor_Mention__c | AE-facing brief: deal health, competitive landscape, technical risks, next steps |

**Knowledge files** (uploaded to the agent, not generated by the data
script): product battlecards, competitor comparison sheets, product-line
one-pagers, a master RFP answer doc.

**Connector**: one CRM connector (MCP-based, per current OpenAI direction)
exposing read access to all 8 CRM objects below; write actions (e.g.
updating a POC status) gated behind an approval step. This phase documents
*how* to configure this — it is not wired to a live system.

## 4. CRM Data Model ("soma-moma")

Eight Salesforce-style objects, in dependency order:

1. **Account** — company being sold to (name, industry, employee count,
   region, tech-stack tags)
2. **Contact** — buyer-side people (name, title, account, role: economic
   buyer / technical evaluator / champion)
3. **Opportunity** — the deal (stage, amount, close date, pod, product
   line, primary competitor)
4. **OpportunityTeamMember** — links an AE and an SE (with role) to an
   Opportunity
5. **POC__c** — POC tracking (opportunity link, start/end dates, success
   criteria, status, environment type)
6. **SE_Activity__c** — discovery calls, demos, RFP work, POC check-ins
   (type, date, SE, notes)
7. **Competitor_Mention__c** — competitor name, opportunity link, context,
   threat level
8. **RFP_Response__c** — question, category, answer text, optional
   opportunity link (nullable — some are reusable master answers not tied
   to one deal)

Fluxora-specific vocabulary used throughout the generated data:

- **Product lines**: Lakehouse Core, PipelineOps, ModelServe, Governance
  Suite
- **Deal stages**: Discovery → Technical Validation → POC → Business Case →
  Negotiation → Closed Won / Closed Lost
- **Competitors**: Databricks, Snowflake, Confluent (real-world names, used
  because this is realistic for the space and the demo is internal/
  non-commercial)

## 5. Data Generation Script

- **File**: `scripts/generate_crm_data.py`
- **Language/tooling**: Python 3 + the `faker` library
- **Reproducibility**: a fixed random seed so re-runs produce identical
  output
- **SE roster**: the script first generates a fixed 42-person SE roster
  (assigned across the 3 pods, with 2 managers + 1 director) before
  generating any other object, so SE ownership is consistent across every
  CSV — an SE appearing on an Opportunity also appears correctly on that
  Opportunity's POC and SE_Activity records.
- **Row volumes** (approximate, larger/realistic tier):

  | File | Rows |
  |---|---|
  | accounts.csv | ~200 |
  | contacts.csv | ~600 |
  | opportunities.csv | ~350 |
  | opportunity_team_members.csv | ~400 |
  | pocs.csv | ~120 |
  | se_activities.csv | ~900 |
  | competitor_mentions.csv | ~150 |
  | rfp_responses.csv | ~70 |

- **Output**: 8 CSV files written to `output/`, one per object, using
  simple prefixed IDs (e.g. `ACC-0001`, `OPP-0001`) as stable keys —
  compatible with Salesforce Data Loader / Bulk API external-ID upsert
  patterns, and equally easy to open in a spreadsheet tool.
- **Referential integrity**: every foreign-key column (e.g. `AccountId` on
  Contact, `OpportunityId` on POC__c) must reference a row that exists in
  the parent file generated earlier in the run.

## 6. Guideline Document (ChatGPT Enterprise setup)

A separate Markdown document (`docs/chatgpt-enterprise-setup-guide.md`)
explaining, for a reader setting this demo up by hand:

1. How to create a Workspace Agent in ChatGPT Enterprise and configure its
   instructions.
2. How to define the four Skills (Discovery Prep, POC Tracker, RFP
   Drafting, Deal Brief) as distinct sub-capabilities of the agent.
3. How to upload knowledge files (battlecards, RFP master doc) and current
   file/size limits.
4. How a CRM Connector would be configured (conceptually — auth model,
   read vs. write scoping, approval gating for writes), without actually
   connecting one.
5. Relevant workspace admin controls (who can create/edit/publish agents,
   domain/connector allowlisting).
6. A short callout on the Custom GPT + Actions fallback path, for
   workspaces without Agents/Skills enabled.
7. An explicit caveat that this feature area is new/evolving and menu
   names may have shifted since this doc was written.

## 7. Out of Scope (this phase)

- Actually creating the Agent, Skills, or Connector inside a real ChatGPT
  Enterprise workspace.
- Connecting the generated CSV data to any live CRM (Salesforce org,
  Airtable, etc.).
- Building any of the four skills' underlying logic/prompts beyond what's
  described in the guideline document.

## 8. Testing / Validation

- Running `scripts/generate_crm_data.py` twice with the same seed produces
  byte-identical output.
- Every foreign key in every generated CSV resolves to an existing row in
  its parent CSV (spot-checked, not necessarily exhaustively asserted in
  code).
- Row counts land within the approximate ranges in Section 5.
