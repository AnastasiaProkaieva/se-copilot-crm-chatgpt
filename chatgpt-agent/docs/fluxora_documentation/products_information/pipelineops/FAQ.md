# PipelineOps — Technical FAQ

## Architecture & Deployment

### What is PipelineOps and how does it fit into the Fluxora platform?
PipelineOps is Fluxora's fully managed, DAG-based orchestration layer, generally available since 2024, purpose-built for data and ML workflows that run on Fluxora's lakehouse storage and Governance Suite. It orchestrates ingestion, transformation, feature engineering, model training, and batch inference pipelines from a single control plane, and captures lineage automatically as those pipelines run — positioning it as the connective tissue between pipeline execution and governance/audit readiness, not a standalone scheduler bolted on afterward. It's scoped specifically to orchestrating pipelines that touch the lakehouse; it is not a general-purpose workflow engine for non-data use cases like CI/CD or business process automation, and it doesn't replace downstream BI scheduling or reverse-ETL tools.

### Do customers need to run or manage any scheduler infrastructure themselves?
No — that's the core operational value proposition. PipelineOps is a Fluxora-hosted, multi-tenant scheduler control plane; customers don't provision, patch, upgrade, or capacity-plan a metastore, scheduler, or worker fleet, in contrast to self-managed Airflow. DAG authoring is code-first (Python and YAML) with a local CLI for validating DAGs before deployment, and standard Git-based CI/CD integration supports promoting DAGs across dev/staging/prod with environment-scoped variables and secrets. Note that DAG authoring today is code-first only — there is no visual, no-code DAG builder aimed at business analysts in the current release.

### What are PipelineOps' scale limits, and how resilient is the control plane to infrastructure failure?
Internally, PipelineOps is designed and load-tested to support DAGs of up to 5,000 tasks and tenant-level concurrent execution of 10,000+ active task instances without control-plane degradation, with event-driven and dependency-triggered task starts within 30 seconds of the trigger condition being met at P95. On resilience, the control plane is designed to tolerate a single-AZ failure without dropping scheduled triggers, backed by a 99.9% uptime SLA for GA customers, with control-plane metadata (DAG definitions, run history, lineage events) backed up to an RPO ≤15 minutes / RTO ≤4 hours target. One honest caveat for RFPs: the 5,000-task/10,000-concurrent-task figures reflect internal load testing, and a large-account stress-testing program validating them against real enterprise workload profiles is in progress now (Q3 2026) — for workloads approaching that ceiling, recommend validating with a POC rather than citing the numbers as an externally benchmarked guarantee.

## Security & Compliance

### How are DAG definitions, parameters, and secrets protected?
All DAG definitions, parameters, and secrets are encrypted at rest and in transit. Secrets management integrates with customer-managed KMS keys (BYOK), which is generally available for enterprise-tier customers. RBAC is enforced at the DAG, task, and connection-credential level, integrated with Fluxora's centralized IAM, so pipeline ownership and edit/execute permissions map cleanly to team boundaries rather than relying on a flat permission model.

### What compliance certifications cover PipelineOps, and how is audit logging handled?
The PipelineOps control plane is within Fluxora's SOC 2 Type II coverage. Full audit logging is maintained for DAG edits, permission changes, and manual run triggers — typically the artifact prospects' security and compliance teams request during procurement review. Multi-tenancy isolation is enforced logically at both the scheduler and metadata-store level, with no shared execution state across tenants.

### How does PipelineOps support customers with strict data-residency requirements?
Lineage and execution logs are retained consistent with customer data-residency configuration today, with no cross-region movement of logs/lineage without explicit customer opt-in. That said, data-residency-aware storage controls specifically for regionally-constrained tenants are on the roadmap (planned for Q4 2026) as a dedicated hardening effort — current DR design assumes a standard regional backup topology. For prospects with hard residency constraints, flag this explicitly to Legal/Compliance and Product during discovery rather than assuming full coverage out of the box.

## Integration & Interoperability

### What compute engines and external systems can PipelineOps orchestrate tasks across?
A single PipelineOps DAG can span Fluxora's lakehouse compute (SQL and Spark jobs), model-serving endpoints, and external systems via connectors — including dbt, REST APIs, and webhook triggers. This cross-engine reach is a deliberate differentiator: rather than confining orchestration logic to a single vendor's compute layer, PipelineOps is meant to be the neutral control plane sitting across whatever engines a customer's pipelines actually touch. Note that PipelineOps orchestrates compute within Fluxora's managed compute pools; it does not orchestrate arbitrary third-party Kubernetes clusters in the current release.

### How does lineage capture work, and does it require manual instrumentation?
Lineage capture is automatic and native — PipelineOps captures task-level input/output dataset references, execution metadata, and run-level lineage graphs, streaming them in real time into Governance Suite without requiring customers to hand-instrument DAGs. Coverage is strongest for tasks running on Fluxora-native compute (lakehouse SQL/Spark, model serving). For tasks executed via external connectors — e.g., arbitrary REST APIs — lineage can be incomplete unless the connector explicitly emits lineage metadata; a connector-level lineage emission SDK to close this gap is planned for Q3 2026, feeding into a broader cross-engine lineage completeness push targeting 99%+ production task-execution lineage coverage. When a prospect's use case leans heavily on custom/external connectors, scope this explicitly during discovery.

### What does the Airflow migration path look like for teams with existing DAGs?
PipelineOps includes a DAG conversion utility that parses existing Airflow DAG definitions and flags or auto-migrates common operators to PipelineOps equivalents, plus a compatibility mode covering a defined subset of Airflow operators. Compatibility-mode coverage has already been expanded to include additional operator mappings and common sensor/custom-operator patterns surfaced during recent enterprise migrations (shipped in Q1–Q2 2026), with further coverage expansion for plugin-dependent and custom-operator DAGs planned for Q3 2026, and a v2 of the conversion utility — aimed at increasing the share of DAGs that auto-migrate without manual remediation — planned for Q4 2026. Because DAG authoring intentionally mirrors the mental model Airflow users already know, the migration pitch to a Head of Data Engineering is: keep the DAG concepts your team knows, shed the infrastructure ownership. For heavily customized or plugin-dependent Airflow estates, set expectations during discovery that some operators may require manual remediation rather than promising full automatic coverage.

## Competitive Differentiation

### How is PipelineOps different from Databricks Workflows?
Databricks Workflows orchestrates well within the Databricks compute environment, but it's inherently tied to that platform's job clusters and notebooks — it isn't designed as a neutral orchestration layer across multiple engines or vendors, and its lineage is scoped to jobs run inside Databricks notebooks. PipelineOps is engine-agnostic: a single control plane orchestrates lakehouse SQL/Spark jobs, model-serving tasks, and third-party connectors, and because it's native to Fluxora's Governance Suite, it produces unified cross-pipeline lineage rather than lineage limited to one vendor's compute layer. For prospects who aren't fully committed to a single-vendor compute stack, this materially reduces long-term lock-in risk relative to standardizing orchestration on Workflows. (For your own awareness: if Databricks extends Workflows with broader cross-engine support or improved lineage in a future release, this differentiation narrows — don't over-anchor a deal exclusively on this point.)

### Why would a team running self-managed Apache Airflow move to PipelineOps?
The differentiation is almost entirely operational, not conceptual — PipelineOps intentionally mirrors the DAG-authoring model Airflow users already know, so migration isn't a re-education project. What changes is who owns the infrastructure: self-managed Airflow requires the customer to run, patch, scale, and upgrade their own scheduler, metastore, and worker fleet, typically a multi-person, ongoing platform-engineering commitment. PipelineOps removes that operational surface as a managed service and adds lineage capture, Governance Suite integration, and enterprise RBAC that self-managed Airflow doesn't provide out of the box — those capabilities typically require bolting on and separately maintaining third-party tooling (e.g., OpenLineage integrations) in an Airflow-only stack. The pitch to a Head of Data Engineering: keep the mental model, shed the infrastructure ownership, get lineage and governance without assembling it yourself.

## Pricing & Licensing

### Is BYOK secrets management available on every PipelineOps tier?
BYOK secrets management, integrated with customer-managed KMS keys, is generally available specifically for enterprise-tier customers. Standard encryption at rest and in transit for DAG definitions, parameters, and secrets applies across the board; BYOK is the enterprise-tier enhancement on top of that baseline.

### Is the Airflow migration tooling bundled into a specific tier, or priced separately?
This is genuinely still being finalized on the product/pricing side — whether Airflow migration tooling and BYOK secrets management are bundled into a single enterprise tier or sold as separate add-ons has not yet been locked down. Don't commit to a specific bundling structure in an RFP response; confirm current packaging with deal desk/Product before finalizing pricing language for a competitive displacement deal, since this can materially affect how you position cost against a self-managed Airflow TCO comparison.

## Roadmap

### Has Fluxora validated the 5,000-task DAG / 10,000+ concurrent task-instance targets against workloads at our scale?
Those targets currently reflect internal load testing, not yet a benchmark independently stress-tested against the largest prospective enterprise accounts' real workloads. A large-account stress-testing program to validate those targets against real enterprise workload profiles is in progress in Q3 2026, with the goal of publishing validated benchmarks usable in competitive RFP responses in Q1 2027. Until that program completes, the responsible approach for a large-scale prospect is to run a POC against their actual DAG shapes and concurrency patterns rather than citing the current targets as an externally validated guarantee.

### When will lineage capture be complete for tasks running through third-party/external connectors?
Today, lineage coverage is strongest for tasks on Fluxora-native compute; external-connector tasks (e.g., arbitrary REST APIs) may have incomplete lineage unless the connector explicitly emits lineage metadata. A connector-level lineage emission SDK is planned for Q3 2026, followed by a broader cross-engine lineage completeness push in Q4 2026 aimed at extending automatic lineage capture to external-connector task types toward the 99%+ production lineage-coverage target. If a prospect's governance requirements hinge on complete lineage across heavily externally-connected pipelines, characterize this as an active area of investment rather than a fully solved problem today.
